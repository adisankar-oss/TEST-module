1. Introduction & Project Goals
1.1 What You Are Building
You are building an 
AI-powered video interview bot
 that joins a scheduled video meeting in place of a human interviewer. The bot is represented by a talking AI avatar, asks intelligent and adaptive questions based on the candidate's resume and the job description, listens to the candidate's spoken answers, evaluates them in real time, and produces a structured interview report at the end.
When a recruiter at Jobsify schedules an AI interview, the candidate receives a meeting link. At the scheduled time, the 
AI avatar bot joins the call as a participant
. The candidate sees a realistic animated interviewer on screen. The interview proceeds naturally — the avatar speaks, listens, asks follow-up questions, and wraps up professionally. The recruiter can monitor live or review the full report afterward.
🎯 Goal: 
A candidate should not need special software — the interview happens in a standard video call (Google Meet, Zoom, or a custom WebRTC room) and feels like talking to a real interviewer.

1.3 Design Principles
Candidate-first:
 The candidate should have a low-friction experience. No app download. No account required. One meeting link.
Adaptive, not scripted:
 The avatar should not robotically read from a fixed list. Questions should respond to what the candidate just said (follow-up on interesting points, skip topics they already answered).
Transparent AI:
 At the start of every interview, the avatar must clearly state: 'I am an AI interviewer. This interview is being recorded and evaluated.' Non-negotiable.
Graceful degradation:
 If any module fails (TTS fails, STT drops, LLM times out), the session continues with safe fallbacks — not a crash.
Recruiter-in-control:
 A recruiter dashboard lets humans monitor live sessions, intervene, extend time, or terminate early.
Non-goals:
 You are NOT building a job board, a resume parser, a scheduling calendar, or a billing system. Those are Jobsify's responsibility. Your API will receive a pre-scheduled session payload and must handle everything from join to report.

3. Module Specifications
M1 — Session Orchestrator
3.1 Objective
M1 is the brain of the system. It manages the interview as a finite state machine (FSM) and coordinates all other modules. It does not do speech, video, or AI itself — it only orchestrates.
3.2 Interview States (FSM)
State
Trigger In
Actions
WAITING
Session scheduled
Bot joins meeting room. Avatar visible but silent. Wait for candidate to join.
INTRO
Candidate detected in call
Avatar plays mandatory AI disclosure + greeting. Reads candidate name and job title from session config.
ASKING
Intro complete OR prior answer received
M5 generates next question. M3 TTS speaks it. M4 animates avatar. Start answer timeout timer.
LISTENING
Avatar finishes speaking
M2 begins capturing candidate audio. M3 STT streams transcript. Detect silence (> 3 seconds) as end-of-answer signal.
EVALUATING
Silence detected OR max_answer_time reached
M6 evaluates the transcript. Score stored. M1 decides: follow-up, next topic, or wrap-up.
FOLLOWUP
Evaluator signals low depth or interesting thread
M5 generates a follow-up question targeting the gap. Returns to ASKING.
WRAPPING
All topics covered OR max_duration reached
Avatar delivers closing script. Thanks candidate. Announces report delivery timeline.
ENDED
Avatar finishes closing
Bot leaves call. M7 triggered async to generate report. Session state = COMPLETE.
ERROR
Any module throws unrecoverable exception
Avatar speaks: 'I experienced a technical issue. Please contact support.' Bot leaves gracefully.

3.6 Daily.co Integration Pattern
# backend/app/meeting_bot/daily_bot.py
from daily import Daily, CallClient

class InterviewBot:
    def __init__(self, room_url: str, avatar_audio_queue: asyncio.Queue):
        self.client = CallClient()
        self.room_url = room_url
        self.avatar_audio_queue = avatar_audio_queue

    async def join(self):
        self.client.join(self.room_url, client_settings={"inputs": {
            "camera": {"isEnabled": True},   # avatar video injected here
            "microphone": {"isEnabled": True}
        }})

    def inject_audio(self, pcm_chunk: bytes):
        # Feed TTS audio into the call as the bot's microphone
        self.client.send_audio(pcm_chunk)

    def on_participant_audio(self, participant_id: str, audio_data: bytes):
        # Forward to STT pipeline
        self.stt_queue.put_nowait(audio_data)

3.8 TTS — Text to Speech
The TTS pipeline converts the question text to audio, streams it to M4 (for lip sync) and to M2 (for audio injection into the call). Always stream — never wait for the full audio to generate.
# ElevenLabs streaming TTS
from elevenlabs.client import ElevenLabs
from elevenlabs import stream

client = ElevenLabs(api_key=settings.elevenlabs_api_key)

async def speak(text: str, voice_id: str) -> AsyncGenerator[bytes, None]:
    audio_stream = client.generate(
        text=text,
        voice=voice_id,
        model="eleven_turbo_v2",  # lowest latency model
        stream=True,
    )
    async for chunk in audio_stream:
        # Fan out: send to avatar engine AND meeting bot simultaneously
        await avatar_queue.put(chunk)
        await bot_audio_queue.put(chunk)

M5 — Question Intelligence Engine
3.13 Objective
M5 generates the next interview question at each ASKING state. Questions must be relevant to the job, informed by the candidate's background, and adaptive to what was already said. This is the most intellectually complex module.
3.14 Question Generation Algorithm
Load context:
 Job description, required skills, candidate resume summary, interview config (topics, max questions, time elapsed), full transcript so far.
Select topic:
 Based on which topics remain uncovered (track covered topics in session state) and interview stage (early = rapport + background; mid = technical; late = behavioural + culture fit).
Generate question with LLM:
 Pass the full context to GPT-4o-mini with a structured prompt. Request JSON output with the question, rationale, expected answer keywords, and follow-up triggers.
Validate output:
 Check: question is in English, is a question (ends with '?'), is not a duplicate of previous questions (cosine similarity check). Retry once on failure.
Store in session:
 Append question to session.questions list with timestamp and topic tag.

M6 — Answer Evaluator
3.17 Objective
M6 scores each candidate answer immediately after the answer is transcribed. Scores are visible to the recruiter on the live dashboard and stored for the final report.
3.18 Scoring Dimensions
Dimension
Max Score
What to Measure
Relevance
25
Does the answer address what was asked? Penalise tangents and non-answers.
Depth
25
Is the answer substantive? Look for: specific examples, numbers, outcomes. Penalise vague generalities.
Technical Accuracy
25
For technical questions: are facts, terminology, and approaches correct? Use expected_keywords from M5 as a weak signal.
Communication
15
Is the answer clear and structured? STAR format (Situation, Task, Action, Result) scores max. Rambling or incoherent answers score low.
Red Flags
10 (deduction)
Detect: contradicts resume, admits critical skill gap, unprofessional language, extremely short answer (< 20 words for non-trivial question).
3.19 Evaluation Prompt Template
SYSTEM: You are an expert interview evaluator. Score candidate answers precisely.
Do not be lenient. Return structured JSON only.

USER:
QUESTION: {question_text}
EXPECTED KEYWORDS: {expected_keywords}
CANDIDATE ANSWER: {answer_transcript}
ROLE LEVEL: {career_stage}

Return JSON:
{
  "relevance_score": 0-25,
  "depth_score": 0-25,
  "technical_score": 0-25,
  "communication_score": 0-15,
  "red_flags": [],  // array of strings, empty if none
  "overall_score": 0-100,
  "brief_feedback": "1-2 sentence summary for recruiter (not shown to candidate)",
  "needs_followup": true | false,
  "followup_reason": "optional: what gap to probe in the follow-up question"
}
⚠️ Latency Constraint: 
Evaluation must complete within 5 seconds. If the LLM is slow, return a partial score with is_preliminary=true and update it when the full result arrives. Never block the next question.
M7 — Interview Report Generator
3.20 Objective
M7 generates a structured post-interview report for the recruiter. The report is generated asynchronously after the session ends (using a Celery task). It must be ready within 3 minutes of session end.
3.21 Report Sections
Candidate Summary:
 Name, job applied for, interview date, duration, avatar persona used.
Overall Score:
 Weighted aggregate of all answer scores. Display as a number (0–100) and a tier: Exceptional (90+), Strong (75–89), Adequate (60–74), Weak (45–59), Not Recommended (<45).
Per-Topic Scores:
 Average score per topic area (technical, behavioural, etc.) as a mini scorecard.
Question-by-Question Breakdown:
 Each question, the answer transcript, score breakdown, and recruiter feedback note.
Strengths:
 Top 3 strengths inferred from high-scoring answers. Cite specific quotes from the transcript.
Concerns:
 Top 3 concerns. Include any red flags detected. Cite evidence.
Hiring Recommendation:
 One of: Advance to Next Round / Hold for Comparison / Reject with Feedback / Reject.
Full Transcript:
 Time-stamped full interview transcript with speaker labels (Alex / Candidate).

  overall_score   INTEGER,
  recommendation  VARCHAR(30),           -- advance | hold | reject | reject_feedback
  created_at      TIMESTAMPTZ DEFAULT NOW()

CREATE TABLE questions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id      UUID REFERENCES sessions(id) ON DELETE CASCADE,
  question_number INTEGER NOT NULL,
  text            TEXT NOT NULL,
  topic           VARCHAR(30) NOT NULL,
  is_followup     BOOLEAN DEFAULT FALSE,
  asked_at        TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE answers (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  question_id      UUID REFERENCES questions(id) ON DELETE CASCADE,
  session_id       UUID REFERENCES sessions(id) ON DELETE CASCADE,
  transcript       TEXT NOT NULL,
  duration_seconds INTEGER,
  relevance_score  INTEGER,
  depth_score      INTEGER,
  technical_score  INTEGER,
  communication_score INTEGER,
  overall_score    INTEGER,
  red_flags        JSONB DEFAULT '[]',
  feedback         TEXT,
  answered_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE transcripts (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    UUID REFERENCES sessions(id) ON DELETE CASCADE,
  speaker       VARCHAR(20) NOT NULL,   -- 'avatar' or 'candidate'
  text          TEXT NOT NULL,
  ts_offset_ms  INTEGER NOT NULL,       -- milliseconds from session start
  is_final      BOOLEAN DEFAULT TRUE
);

CREATE TABLE reports (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id    UUID UNIQUE REFERENCES sessions(id) ON DELETE CASCADE,

11. Frequently Asked Questions
Q: Does the candidate need to download anything?
No. They receive a standard meeting link (Daily.co room URL in development, Google Meet / Zoom in production). They join exactly as they would any normal video call. The bot appears as another participant.
Q: What if the candidate's internet drops mid-interview?
M1 FSM should transition to a PAUSED state when the candidate's participant track is lost (Daily.co fires a participant-left event). Wait up to 3 minutes for reconnect. If no reconnect, transition to ENDED with reason='candidate_disconnected'. The partial report should still be generated.
Q: What if the D-ID quota runs out?
This is handled by the avatar fallback rule in M4. A static profile image with an animated audio waveform overlay must always be the fallback. The interview continues without interruption.
Q: Can the LLM generate inappropriate questions?
This risk must be mitigated by your prompt design. The prompt must explicitly forbid questions about: age, gender, marital status, religion, national origin, disability (all protected characteristics). Add a validation step: run the generated question through a keyword blocklist before speaking it. If blocked, pull from the question bank instead.
Q: How many questions per interview?
Configurable per session via max_questions (default 10, max 20). The FSM tracks covered topics and wraps up gracefully when all topics are covered or the time limit is hit — whichever comes first.
Q: Do we need a mobile-friendly recruiter dashboard?
Responsive layout is expected (Tailwind makes this easy) but deep mobile optimization is not required for the internship. Prioritise desktop-first.
Q: Can we swap D-ID for another avatar provider?
Yes. HeyGen is a strong alternative. Synthesia is higher quality but more expensive. The M4 module should be written with an abstract 
AvatarEngineBase
 class so the provider can be swapped by changing an env var.
